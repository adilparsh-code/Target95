# Target95 AI Development Rules

## Project Overview

Target95 is a production-grade AI Powered Learning Platform for ICSE & ISC Computer Science students.

Technology Stack

- Next.js 15 (App Router)
- React
- Tailwind CSS
- JavaScript
- Firebase (Future)
- Vercel Deployment

---

## Mission

Build clean, scalable, reusable and production-ready software.

Never write demo code.

Everything should be suitable for deployment.

---

## Development Workflow

Before writing code:

1. Understand existing architecture.
2. Reuse existing components.
3. Search for duplicate functionality.
4. Plan implementation.
5. Implement.
6. Test.
7. Refactor if needed.

Never skip these steps.

---

## File Rules

Never assume folders exist.

Always inspect the existing workspace.

Prefer modifying existing files.

Create new files only when necessary.

Do not duplicate components.

---

## Code Quality

Always

- Functional Components
- Reusable Components
- Small Components
- Clean Imports
- Meaningful Variable Names
- Consistent Formatting

Avoid

- Duplicate Code
- Huge Components
- Hardcoded Values
- Inline Styles

---

## UI Rules

Every screen must be

- Responsive
- Mobile First
- Accessible
- Dark Mode Ready
- Consistent
- Modern
- Professional

---

## Architecture

Business Logic

↓

Hooks

↓

Reusable Components

↓

Pages

Never put business logic inside page files.

---

## Safety Rules

Do NOT

- Delete existing files
- Rename folders
- Install packages
- Change architecture

Unless explicitly instructed.

---

## Approval Rules

Do NOT ask approval for

- Creating Components
- Editing Components
- Tailwind Changes
- JSX Changes
- Refactoring

Ask approval only for

- Installing Packages
- Deleting Files
- Database Changes
- Firebase Changes
- Architecture Changes

---

## AI Behaviour

Never stop because of assumptions.

Never say

"I cannot continue."

Instead

- Investigate
- Explain
- Suggest alternatives
- Continue

If terminal commands fail

Use editor APIs.

---

## Completion Rules

A feature is complete only if

- Code is clean
- Responsive
- Reusable
- No build errors
- No console errors

Then provide

- Files Modified
- Components Created
- Summary
- Future Improvements

---

## Coding Style

Think like

Senior Software Engineer

NOT

Code Generator.

## Deletion Rule

Never delete any file based on assumptions.

Before deleting any file:

1. Search the entire project for references.
2. Confirm there are zero usages.
3. Report all findings.
4. Delete only after confirmation.

Never delete files automatically.
